# 项目适配指南

> ultra-plan 是项目无关的通用 skill。项目特定的约定（API 风格、env 管理、响应格式、测试标准、目录布局、截图位置等）由主 agent 在 Phase 0.1 自动**读取项目级配置文件**抽取后注入到生成的 md 中。

---

## 1. 主 agent 该读哪些项目级配置

主 agent 进入 ultra-plan 时，按以下优先级**逐一检查并读取**（用 Read 工具，不派 subagent）：

### 一级（强制读，存在即必读）

| 文件 | 含义 | 抽取重点 |
|---|---|---|
| `CLAUDE.md` | Claude Code 项目指令 | 全部约定 |
| `.claude/CLAUDE.md` | Claude Code 项目指令（备选位置）| 全部约定 |
| `AGENTS.md` | 通用 agent 指令（OpenAI / Cursor / Aider 等共用）| 全部约定 |

### 二级（存在则读，补充信息）

| 文件 | 含义 | 抽取重点 |
|---|---|---|
| `.cursorrules` | Cursor 项目规范 | 代码风格、命名、目录约定 |
| `.cursor/rules/*.mdc` | Cursor 新版分文件规范 | 同上 |
| `README.md` | 项目说明 | 顶部的"开发约定 / 贡献指南"段 |
| `CONTRIBUTING.md` | 贡献指南 | 测试要求、提交规范 |
| `.github/copilot-instructions.md` | Copilot 项目指令 | 同上 |

### 三级（用于推断技术栈）

| 文件 | 用途 |
|---|---|
| `package.json` | 推断 JS/TS 框架（Next / Vite / etc）、scripts |
| `pyproject.toml` / `requirements.txt` | 推断 Python 框架（FastAPI / Django / etc）|
| `go.mod` / `Cargo.toml` / `Gemfile` | 推断其他语言栈 |
| `tsconfig.json` | TS 配置 |

### 四级（已有项目特定 skill）

如果项目根有 `.claude/skills/<name>/`，把项目级 skill 的 SKILL.md 也读一遍——这些通常是项目设计语言、特定模块约定等。

## 2. 抽取项目级红线的标准类目

主 agent 把读到的内容按以下 8 个类目归类，每个类目找出该项目的硬约定：

### 2.1 API 风格

- HTTP method 偏好（RESTful / RPC / GraphQL）
- URL 格式约定
- 路径前缀（`/api/v1/` / `/api/` / 无前缀）
- 错误响应格式
- OpenAPI / 文档生成约定

### 2.2 响应格式

- 统一响应信封（`{code, data, message}` / `{success, data, error}` / 直接返回）
- 错误码规范
- 分页响应格式

### 2.3 env / 配置管理

- env 文件位置（`.env` / `.env.local` / `config/*.yaml` / `env/*.json` 等，按项目约定）
- 是否禁止 `.env` 散落在子项目
- 启动命令（`pnpm dev` / `pnpm start:*` / `make dev` / etc）

### 2.4 目录约定

- 后端目录布局（`src/` / `app/` / `apps/<service>/`）
- 前端目录布局
- 测试目录位置
- 文档 / 规划目录（`.todolist/` / `docs/plans/` / `notes/`）

### 2.5 测试约定

- 必跑测试命令
- 覆盖率阈值
- E2E 测试工具（Playwright / Cypress / Selenium）
- 何时跑测试（每次提交 / PR / 发布前）

### 2.6 数据库 / 存储

- ORM / 查询库
- 迁移工具
- 连接池约定（如 Supabase / PgBouncer 的特殊配置）
- 是否硬约定使用某些托管服务（Supabase / Firebase / Auth0 等）

### 2.7 资源 / 产物落盘

- 截图位置
- 日志位置
- 临时文件位置
- 构建产物位置

### 2.8 安全 / 合规

- 加密字段约定
- 审计日志要求
- JWT / 会话管理
- 多租户隔离（RLS / 应用层）

## 3. 注入到生成 md 的位置

抽取后的红线注入到以下两个位置：

### 3.1 `00_总体规划.md` §5.3 派发前必须强调

**通用条目**（任何项目都贴）：
```
- 严格遵循项目级 CLAUDE.md / AGENTS.md 全部约定
- 不要修改项目级配置文件本身
- 改 endpoint 后跑项目要求的文档生成命令
```

**项目特定条目**（按 §2 八个类目从项目配置抽取）：
```
- API 风格：<从 §2.1 抽取>
- 响应格式：<从 §2.2 抽取>
- env 管理：<从 §2.3 抽取>
- 截图位置：<从 §2.7 抽取>
- 测试要求：<从 §2.5 抽取>
- ...
```

### 3.2 每个模块 md §8 派发前必须强调

模块 md 的 §8 = `00_总体规划.md §5.3` 全文复制 + 该模块特有的红线（如有）。

模块特有红线举例：
- "本模块涉及隐私字段，必须 AES-GCM 加密"
- "本模块是管理后台，admin JWT ≠ 用户 JWT"
- "本模块涉及大表分页，禁用 OFFSET"

## 4. 项目无 CLAUDE.md 的兜底

如果项目根没有任何项目级配置：

1. **不**编造红线（不要从其他项目带过来）
2. `00_总体规划.md §5.3` 只贴通用条目
3. 模块 md §8 只贴通用条目
4. 在 `00_总体规划.md §0 背景` 末尾加一段提示：

```
⚠️ 注意：本项目根目录未发现 CLAUDE.md / AGENTS.md / .cursorrules 等项目级配置。
ultra-plan 已使用通用模板生成。
建议在执行前补一份项目级约定文档（推荐 CLAUDE.md），否则各 subagent 实现时风格易飘。
```

## 5. 抽取流程示例

### 假想项目 A：有完善 CLAUDE.md

主 agent 读 CLAUDE.md，看到：
- "API 用 RESTful，统一 /api/v2/ 前缀"
- "响应统一 `{success: bool, data: any, error?: string}`"
- "env 走 .env.local，禁 .env.example"
- "测试用 Vitest，覆盖率 ≥ 70%"
- "截图存 tests/screenshots/"

→ 注入到生成的 md：
```
- API 风格：RESTful，前缀 /api/v2/
- 响应格式：{success, data, error?} 不变
- env 走 .env.local，禁 .env.example
- 测试：Vitest 覆盖率 ≥ 70%
- 截图：tests/screenshots/
```

### 假想项目 B：只有简单 README

主 agent 读 README，看到：
- "项目用 Next 14 + Prisma + tRPC"

→ 注入到生成的 md：
```
- 框架：Next 14 + Prisma + tRPC（按 README）
```

其他类目无明确约定 → 不注入（不编造）。

### 假想项目 C：完全无配置

→ 注入到生成的 md：只贴通用条目 + 在 §0 加提示用户补 CLAUDE.md。

## 6. 反模式

- ❌ 把其他项目的约定 / 训练数据里的"行业默认"带入当前项目
- ❌ 凭印象 / 训练数据猜项目约定（必须从配置文件读）
- ❌ 项目无 CLAUDE.md 时凭"行业最佳实践"编造红线
- ❌ 把 Phase 0.1 项目探测派给 subagent 做（主 agent 自己读，避免上下文丢失）
- ❌ 抽取后不复述给用户确认就直接写进 md（建议在 Phase 0 弹窗里附带"已识别的项目约定摘要"，让用户确认或更正）

## 7. Phase 0 项目探测附加问题（推荐）

在 Phase 0 弹窗的 4 个标准问题之后，**如果项目级配置抽取出超过 5 条红线**，可以追加一个问题：

```
我从 <CLAUDE.md / AGENTS.md / etc> 抽取到以下项目约定：
- <红线 1>
- <红线 2>
- ...

要全部注入到 ultra-plan 生成的 md 里吗？
(A) 全部注入 (Recommended)
(B) 让我看看哪些不要
(C) 都不要，纯通用模板
```

这样用户可以纠正过期 / 错误的约定。
