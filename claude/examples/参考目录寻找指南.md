# 参考目录寻找指南

> ultra-plan 不内嵌任何项目特定的 example 文件（避免给其他项目带入错误约定）。本文档说明主 agent 在 Phase 2 / Phase 3 时**如何在当前项目里**找到合适的参考目录，仿造结构写本次的 ultra-plan 文档。

---

## 1. 在哪里找参考

### 1.1 项目自身的历史 plan / 规划目录

主 agent 在 Phase 2 拆解需求时，先扫描项目根：

```bash
ls -d .todolist/* docs/plans/* docs/rfcs/* notes/plans/* 2>/dev/null
```

常见命名约定：
- `.todolist/<feature>-v<N>/`
- `docs/plans/<feature>/`
- `docs/rfcs/NNN-<feature>.md`
- `notes/plans/<date>-<feature>/`
- `.cursor/notes/`
- `.claude/plans/`

如果找到现有规划目录 → **优先仿造现有结构**（保持与项目一致比"模板更好"重要）。

### 1.2 项目的 README / CONTRIBUTING

`README.md` 顶部、`CONTRIBUTING.md` 经常会写明项目对"新功能 / 大需求"的规范流程（如 RFC 流程、ADR 流程）。

### 1.3 项目级 skill

若项目根有 `.claude/skills/<name>/`，里面可能有项目自己的 plan 模板。

---

## 2. 怎么用找到的参考

主 agent 在 Phase 2 时：

1. **判定本次需求类型**：
   - 日常迭代（在已有功能上加东西）
   - 全新业务域（新 service / 新模块树）
   - 纯前端重构 / 纯后端基建
   - 技术债 / 迁移类
   - 跨产品线集成

2. **读 1-2 个最相似类型的现有规划目录**（用 Read 工具，不派 subagent，主 agent 自己看）。重点观察：
   - 章节顺序
   - 模块清单表的列字段
   - 派发指令段的写法
   - 派发前红线的颗粒度
   - 术语统一程度

3. **仿造结构**写本次 00_总体规划.md。**不要照抄业务内容**，照抄结构。

---

## 3. 项目无现成规划目录的兜底

如果项目从来没有过 ultra-plan 风格的规划目录：

- 直接用 ultra-plan skill 的 `templates/` 下的标准模板
- 在 Phase 0 弹窗里建议用户用 `<project_root>/.todolist/<feature>-v<N>/` 作为输出位置（开第一个）
- 在 00_总体规划.md §0 末尾加一段：
  ```
  📝 这是本项目第一份 ultra-plan 风格规划。后续大型需求建议沿用此目录布局。
  ```

---

## 4. 反模式

- ❌ 派 subagent 去读参考目录后总结返回 → 浪费上下文。主 agent 自己读最快
- ❌ 把参考目录的整段内容复制到新 plan → 应该按本次需求重写
- ❌ 看完参考觉得"我的需求太简单不需要全套" → 那就退回普通 `/plan`，不要用 ultra-plan
- ❌ 跨项目搬参考（"我在 X 项目见过 ultra-plan 怎么写，照那个抄"）→ 项目约定不一定一致，照抄会冲突
